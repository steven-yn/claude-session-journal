## 문서 작성 가이드

- 내용을 요약, 정리를 해야할땐 하되 누락된 항목은 없어야합니다.
- 'n줄 이내로 정리' 하는것보다 '누락 항목 없음'이 더 중요한 목표입니다.
- 상세 설명은 중요하지 않거나 너무 많을경우 건너뛰거나 핵심만 요약하세요.
- 너무 비약적인 요약보다는 상세 설명을 줄이고 큰 관점에서 어떤것들을 했는가를 중점으로 작성하세요.
- **중복 취합**: 여러 세션이나 기존 저널에서 동일한 주제/작업이 반복될 경우, 하나로 통합하세요. 세부 구현 내용 반복을 줄이고 큰 단위(목적, 결과, 배움)만 남겨서 요약합니다.
- 섹션별로 N개를 작성하는것에 얽메이지 마세요. 중요한건 사용자가 납득 가능한 내용 이어야 해요.

## 작성 프로세스

**반드시 5단계 작성 프로세스를 따르세요** (journal-writing 스킬의 Writing Process 참조):

1. **항목 분류**: 세션 데이터에서 별개의 주제/작업/학습을 독립 항목으로 분리
2. **누락 검토**: 모든 세션, 수정 파일, 쿼리가 빠짐없이 반영되었는지 확인
3. **내용 작성**: 각 항목에 인사이트 중심으로 살 붙이기
4. **사실 검증**: 세션 데이터에 근거 없는 내용 삭제, 기술명/경로/수치 정확성 확인
5. **가독성 향상**: 블로그 재활용 가능한 문체로 다듬기

## 회고 양식

회고 섹션은 **KPTA(Keep-Problem-Try-Action) 회고 프레임워크**를 따릅니다.
상세 작성 기준과 관점별 가이드는 `kpta-retrospective` 스킬을 참조하세요.

- **Keep**: 유지할 것 — 효과적이었던 practice
- **Problem**: 문제점 — 어려움과 원인 분석
- **Try**: 시도할 것 — 과거 기록 기반으로 제안/추천, 근거 없으면 비워둠
- **Action**: 실행 계획 — 과거 기록 기반으로 제안/추천, 근거 없으면 비워둠

> Try와 Action은 상상해서 채우지 않습니다. 세션 데이터와 과거 저널에서 근거를 찾을 수 있을 때만 제안하고, 최종 판단은 사용자가 합니다.

## 설정 반영

- **코드 스니펫**: `include_code_snippets`가 true이면, 핵심적인 코드만 `max_code_snippet_lines` 이내로 포함
- **토큰 통계**: `include_token_stats`가 true이면 통계 섹션 포함
- **태그**: `tags_auto_generate`가 true이면 작업 내용 기반 태그 자동 생성
- **YAML frontmatter**: `blog_frontmatter`가 true이면 포함

세션 데이터가 없으면 해당 날짜에 Claude Code 세션이 없다고 알려주세요.
